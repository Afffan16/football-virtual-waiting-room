## Purpose

This document estimates the operational cost of running the Football Virtual Waiting Room, to show that the serverless architecture from [`07-system-architecture.md`](07-system-architecture.md) is scalable *and* cost-efficient, built on AWS managed services and DynamoDB's consumption-based pricing.

> **Disclaimer:** the figures here are estimates for architectural planning and comparison, not a quote. Actual AWS costs depend on region, request volume, data size, concurrency, logging, and current AWS pricing. Check the official AWS Pricing pages before any production deployment.

---

## Cost Drivers

Operational cost comes almost entirely from:

- Amazon DynamoDB
- AWS Lambda
- Amazon API Gateway
- Amazon CloudWatch
- Data transfer

Because the solution is entirely serverless, there's no always-running compute sitting idle between ticket releases — cost tracks usage, not uptime.

---

## DynamoDB

### Billing Mode

**On-Demand Capacity Mode**, chosen because it:

- Scales automatically with traffic
- Requires no upfront capacity planning
- Fits unpredictable spikes like a ticket-release moment
- Charges only for requests actually processed

### Estimated Workload

A representative football-match launch:

| Metric | Value |
|---|---|
| Queue Registrations | 1,000,000 |
| Queue Status Requests | 20,000,000 |
| Token Validations | 250,000 |
| Event Lookups | 500,000 |
| Statistics Queries | 100,000 |

This split mirrors the read-heavy traffic distribution modeled in [`03-access-patterns.md#expected-request-distribution`](03-access-patterns.md#expected-request-distribution) — queue status polling dominates by a wide margin.

### DynamoDB Operations

**Write operations**, generated by: queue registration · queue admission · token creation · status updates.

**Read operations**, generated by: queue status · event lookup · token validation · statistics.

### Storage

Stored entities: events, queue entries, admission tokens, statistics, and sessions — the same six item types defined in [`04-data-model.md`](04-data-model.md). TTL automatically clears out tokens and sessions once they expire, which keeps storage cost from growing unbounded over time.

## AWS Lambda

Lambda pricing depends on invocation count, execution duration, and memory allocation. The functions in this system are built to be cheap on all three fronts: short execution time, fully stateless, and a minimal memory footprint per invocation — see the per-function responsibilities in [`00-project-status.md#lambda-responsibilities`](00-project-status.md#lambda-responsibilities).

---

## API Gateway

Every API request is billed. The dominant contributors are the same ones driving DynamoDB read volume: join queue, queue status, validate token, and event lookup. Since queue-status polling generates the largest share of overall traffic, it's also the largest single line item in API Gateway cost.

---

## CloudWatch

CloudWatch cost comes from log ingestion, log storage, custom metrics, dashboards, and alarms. Structured logging (see [`00-project-status.md#coding-standards`](00-project-status.md#coding-standards)) is used deliberately here — enough detail to diagnose issues, without generating log volume that becomes a cost driver in its own right.

## Cost Optimization Techniques

### On-Demand Billing

Avoids paying for idle capacity between events — a meaningful saving for a workload that's essentially silent most of the time and then spikes hard.

### TTL

Automatically deletes expired sessions and tokens, cutting storage cost without a single scheduled cleanup job. Full reasoning: [`04-data-model.md#time-to-live-ttl`](04-data-model.md#time-to-live-ttl).

### Minimal GSIs

Only the indexes that serve a real access pattern exist — GSI1 and GSI2, with GSI3 optional. Fewer indexes means lower storage, lower write amplification, and lower indexing cost. Full reasoning: [`06-index-design.md#why-not-more-gsis`](06-index-design.md#why-not-more-gsis).

### Query-Only Design

Every access pattern uses `GetItem` or `Query` — never a scan. Scans are both slower and more expensive at this scale, so avoiding them entirely isn't just a performance choice, it's a cost one.

### Small Item Size

Only the attributes each item actually needs are stored, which keeps storage cost down and both reads and writes faster.

### Efficient Logging

Logs capture what's operationally useful without defaulting to verbose output in production.

## Cost Comparison

| Architecture | Relative Cost | Operational Overhead |
|---|---|---|
| Traditional Servers | High | High |
| Containers | Medium | Medium |
| Serverless (this design) | Low–Medium | Low |

The serverless approach minimizes idle infrastructure cost while still scaling automatically the moment demand shows up.

## Monitoring Operational Costs

Recommended metrics to track over time: Lambda invocations, Lambda duration, DynamoDB request metrics, API Gateway requests, and CloudWatch log growth. AWS Budgets and billing alarms should be configured to catch unexpected spending before it compounds.

---

## Future Cost Optimizations

- Push-based updates (WebSockets or Server-Sent Events) to cut down on polling volume — the single largest cost driver identified above
- Attribute projection tuning on the GSIs
- Write sharding for events with extreme registration spikes (see [`04-data-model.md#sharding-strategy`](04-data-model.md#sharding-strategy))
- Log retention policies
- Archiving historical queue data instead of keeping it live in the primary table

---

## Summary

The architecture ties cost directly to actual usage through a fully managed, serverless design. The key cost-saving decisions: no always-on servers, automatic scaling, DynamoDB On-Demand billing, automatic TTL cleanup, minimal GSIs, and a query-first data model throughout.

Together, these let the system absorb an unpredictable, spiky workload like a football ticket launch without paying for capacity it isn't using the rest of the time.
